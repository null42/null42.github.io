# UPS 30 天入职训练系统设计

## 背景

当前 `电源` 文件夹已经包含 30 节 UPS 电源软件开发课程、学习记录、资源清单和术语表。它帮助学习者通过了 UPS 企业的软件工程师面试，但现有体系仍偏向面试准备和知识讲解，不能充分支撑入职后的真实工作能力提升。

学习者背景：

- 电气工程专业，学过 UPS 相关专业课，但很多基础已经遗忘或掌握不牢。
- 有电机控制经验，这是通过面试的重要原因。
- 入职后大概率偏向 UPS 控制相关工作，但不能只掌握控制，还需要理解电路拓扑、硬件、嵌入式、系统保护和工程调试。
- 入职后预计会有老员工导师，可用于校准工程理解、确认公司真实技术栈和请教难点。
- 两天后入职，前一个月预计不会承担很难的任务，需要利用这段缓冲期系统补基础。
- 当前可用工具为 Python 和 MATLAB/Simulink，暂时没有实物硬件条件，因此训练必须以仿真和代码验证为主。

## 目标

将当前资料库从“面试导向课件库”升级为“UPS 入职 30 天训练系统”。系统必须帮助学习者在 1 个月内形成以下能力：

1. 看懂 UPS 控制相关文档、框图、代码和仿真模型。
2. 理解 UPS 关键电路拓扑、采样调理、控制算法、嵌入式实时任务和保护逻辑。
3. 能用 Python 和 Simulink 对关键概念进行仿真验证，而不是只停留在文字理解。
4. 能把电机控制经验迁移到 PFC、逆变器、PLL、dq 坐标系、电流环和 SVPWM 等 UPS 控制问题。
5. 能形成可检查的学习产物，包括仿真模型、波形截图、参数表、代码片段、调试记录和每周复盘。

## 非目标

- 不追求 30 天内达到独立设计完整商用 UPS 的水平。
- 不继续把既有 HTML 课程作为主学习资料。
- 不把所有专业课按教材顺序完整重学。
- 不依赖真实硬件、示波器、功率板或公司内部资料。
- 不把学习记录写成泛泛的心得，而是要求有可验证的技术产物。

## 核心原则

### 1. 每个概念必须解释清楚

所有新增概念文档和课程改造都必须使用固定解释结构：

1. 它是什么：用一句不堆术语的话定义概念。
2. 为什么需要它：说明它解决 UPS 或电源控制里的哪个实际问题。
3. 物理直觉：用电压、电流、能量、相位、频率、开关状态或采样时序解释。
4. 数学形式：给出公式，并解释每个变量、单位、正负号和适用条件。
5. 仿真观察：说明在 Python 或 Simulink 里应该看到什么波形、指标或异常现象。
6. 工程落地：说明它在固件中对应哪个模块、变量、ISR、任务、状态机、参数或保护逻辑。
7. 常见误解：列出容易误解的说法，并给出正确判断。

任何只给定义、不解释来源和用途的内容都视为不合格。

### 2. 仿真驱动学习

每个核心主题都必须绑定至少一个仿真任务。仿真任务必须包含：

- 输入条件：参数、拓扑、采样频率、开关频率、负载条件。
- 观察目标：需要看的波形、指标和现象。
- 对照问题：改变一个参数后应该发生什么变化。
- 验收标准：用清晰指标判断是否完成。
- 复盘要求：记录现象、原因、错误尝试和下一步。

Python 主要用于算法原型、离散控制、数据处理和自动化检查。Simulink 主要用于拓扑级、系统级和闭环控制仿真。

### 3. 控制为主线，硬件和嵌入式为支撑

学习顺序不能变成纯硬件课或纯控制课。每个控制主题都要回答三类问题：

- 硬件问题：控制对象是什么，电压电流从哪里来，传感器如何采样。
- 控制问题：参考量如何生成，误差如何形成，控制器如何输出占空比或调制量。
- 固件问题：代码在哪个周期运行，数据如何同步，保护如何打断控制。

### 4. 从电机控制经验迁移

资料库应主动建立电机控制到 UPS 控制的映射：

- FOC 电流环 -> PFC/逆变器电流环。
- 电机 SVPWM -> 逆变器和三电平 SVPWM。
- 转子坐标 dq -> 电网同步 dq。
- 位置观测/锁相思想 -> 电网 PLL。
- 母线电压和电流采样 -> UPS 采样调理和保护。

迁移时必须说明相同点和不同点，不能只说“类似”。

### 5. 每周必须有可验收产物

每周结束时必须至少形成：

- 一组仿真模型或脚本。
- 一份波形和指标记录。
- 一份概念复盘。
- 一份工程问题清单。

### 6. 导师协作必须问题化

入职后如果有导师，不能只问“这个怎么学”或“我哪里不懂”。每次请教前必须准备：

- 背景：当前学习的主题和项目。
- 已验证内容：仿真模型、波形、参数或代码片段。
- 具体困惑：卡在物理直觉、数学推导、仿真现象、固件落地还是公司平台差异。
- 期望确认：希望导师判断哪件事，例如控制框图是否合理、参数量级是否像真实项目、保护策略是否符合工程习惯。
- 记录结果：导师给出的判断、经验规则、公司内部习惯和后续验证任务。

导师反馈需要沉淀到 `mentor-notes/`，但不能记录公司保密资料、内部代码、未公开产品细节或敏感参数。

## 30 天训练路线

### 第 1 周：基础补齐和仿真工具启动

目标：恢复电路、电力电子、采样、PWM 和离散控制基础，建立 Python/Simulink 仿真工作流。

主题：

- KCL/KVL、RLC、一阶/二阶响应。
- Buck、Boost、Buck-Boost、CCM/DCM/BCM。
- MOSFET/IGBT/SiC、二极管、死区、驱动和损耗。
- PWM、SPWM、采样频率、开关频率、ADC 同步采样。
- 连续 PI 到离散 PI。

产物：

- Python：离散 PI 控制器脚本。
- Simulink：Buck 或 Boost 开环模型。
- 复盘：解释 CCM/DCM、占空比、电感电流纹波和采样延迟。

### 第 2 周：UPS 控制核心

目标：把电机控制经验迁移到 UPS 控制，重点掌握 PFC、PLL、逆变器和 PR/PI 控制。

主题：

- 单相 Boost PFC。
- 电压外环、电流内环、输入电压前馈、V^2 RMS 前馈。
- SRF-PLL、单相 PLL 的虚拟正交信号问题。
- 单相逆变器、LC 滤波器、PR 控制。
- 三相 dq 控制和 SVPWM。

产物：

- Simulink：单相 PFC 闭环模型或简化平均模型。
- Python：PLL 或 PR 控制器离散化验证。
- 复盘：解释 PFC 为什么电压环要慢、PLL 为什么会影响并网/同步、PR 为什么适合单相交流稳压。

### 第 3 周：UPS 系统和保护

目标：从单个控制环扩展到 UPS 整机系统，理解模式切换、DC 母线、电池、旁路和故障处理。

主题：

- 在线式 UPS 双变换架构。
- PFC、DC-DC、逆变器之间的母线协调。
- 电池充放电、双向 DC-DC、SOC 基本概念。
- 旁路、维修旁路、静态开关和切换时序。
- 过压、欠压、过流、短路、过温、采样异常。

产物：

- Simulink：简化在线式 UPS 能量流模型。
- Python：UPS 状态机和故障注入脚本。
- 复盘：解释“保护不是简单停机”，而是分级、降额、旁路、锁存和恢复策略。

### 第 4 周：嵌入式工程化

目标：理解 UPS 控制算法如何落到固件架构、任务调度、中断、通信和调试流程中。

主题：

- 控制 ISR、ADC 触发、PWM 更新、Trip-Zone 或硬件保护。
- 前后台系统和 RTOS 任务划分。
- 参数管理、Flash 存储、CRC、默认值恢复。
- Modbus/CAN/SNMP 的工程定位。
- 日志、故障记录、调试流程和版本管理。

产物：

- C 或伪 C：控制 ISR + 状态机 + 保护框架骨架。
- Python：参数表校验或故障日志解析脚本。
- 复盘：解释为什么控制闭环要在 ISR 中运行，为什么通信不能阻塞控制。

## 项目设计

### 项目 1：Boost 基础模型

目的：从电感电流和电容电压理解 Boost 的物理本质。

验收：

- 能解释开关导通/关断两个阶段的电流路径。
- 能观察占空比变化对输出电压和电感纹波的影响。
- 能区分 CCM 和 DCM。

### 项目 2：单相 PFC

目的：理解 UPS 输入级控制。

验收：

- 能解释为什么输入电流要跟随输入电压。
- 能搭建平均电流模式控制框图。
- 能观察电压环带宽过高导致电流畸变的现象。
- 能解释 V^2 RMS 前馈的作用。

### 项目 3：PLL 电网同步

目的：理解并网、旁路同步和 dq 控制的基础。

验收：

- 能解释 PLL 的输入、输出和误差信号。
- 能观察相位阶跃、频率偏差和谐波对 PLL 的影响。
- 能说明 PLL 带宽过高和过低的代价。

### 项目 4：单相逆变器

目的：理解 UPS 输出稳压。

验收：

- 能解释 LC 滤波器如何把 PWM 转成正弦电压。
- 能比较 PI 和 PR 在交流稳压中的差异。
- 能观察负载突变对输出电压的影响。

### 项目 5：UPS 状态机

目的：理解整机模式和故障处理。

验收：

- 能画出市电正常、电池供电、旁路、故障、恢复的状态图。
- 能用 Python 脚本模拟输入条件变化和状态迁移。
- 能区分保护动作、故障锁存、告警和自动恢复。

### 项目 6：固件控制骨架

目的：把控制算法、采样、保护、任务调度和通信连接起来。

验收：

- 能写出控制 ISR 的执行顺序。
- 能区分快环、慢环、状态机和通信任务。
- 能说明哪些保护必须硬件关断，哪些可以软件处理。

## 旧课程归档策略

现有 30 节 HTML 课程不再重组为主线，原因：

- 内容一次性生成痕迹明显，概念解释深度不稳定。
- 图表存在错位和渲染问题，可能影响对拓扑、控制框图和时序的理解。
- 部分内容偏面试表达，不适合作为入职后能力建设的权威材料。
- HTML 课件不便于持续维护、版本对比、仿真关联和学习复盘。

处理方式：

- 将 `lessons/` 移动到 `archive/old-lessons/`。
- 将 `learning-records/` 移动到 `archive/old-learning-records/`。
- 不删除旧内容，避免丢失已有素材。
- 新体系以 `concepts/`、`simulations/`、`projects/`、`debug-records/` 和 `weekly-reviews/` 为主。
- 旧课程中的内容只有在经过重新查证、重写和仿真验证后，才能进入新概念文档或项目文档。

后续新增 `roadmap/lesson-priority-map.md` 不再映射旧课程序号，而是映射新训练模块：

- 学习主题。
- 关键概念。
- 对应仿真任务。
- 对应项目产物。
- 推荐外部资源。
- 可向导师确认的问题。

## 资源策略

资源分三级：

1. 工程主资源：TI、ST、Infineon、Microchip、ADI 等厂商应用笔记和参考设计。
2. 理论主资源：教材、标准、经典控制理论和电力电子资料。
3. 辅助资源：博客、论坛、视频和问答，只用于辅助理解，不能作为唯一依据。

每个概念文档必须尽量链接到工程主资源或理论主资源。涉及当前标准、芯片资料或厂商方案时，需要查证最新资料。

## 导师反馈策略

导师是获取工程智慧的重要来源，但导师时间有限，因此请教必须围绕具体产物进行。

每周至少准备一次导师问题清单，优先问以下类型问题：

- 公司实际 UPS 控制软件中，哪些模块最常见、最重要。
- 公司常用 MCU、DSP、RTOS、仿真工具和调试工具是什么。
- PFC、逆变器、PLL、保护、旁路切换等模块在真实项目中最容易出问题的地方。
- 仿真模型和真实硬件之间通常有哪些差异。
- 控制参数调试时有哪些经验量级和安全边界。
- 新人最应该先读哪些内部文档或代码模块。

禁止记录：

- 公司内部源代码。
- 未公开产品方案。
- 具体客户项目参数。
- 内部测试数据。
- 任何导师明确不能外传的信息。

## 学习记录规范

新学习记录不再只写“关键洞察”，必须包含：

- 今天完成了什么。
- 哪个概念原来不懂，现在如何理解。
- 仿真模型或脚本路径。
- 关键参数。
- 观察到的波形或指标。
- 出现的问题。
- 错误原因。
- 下一步要验证什么。

## 成功标准

30 天后，学习者应能完成以下任务：

1. 独立解释在线式 UPS 的主功率路径、控制路径和保护路径。
2. 搭建并解释 Boost/PFC/PLL/单相逆变器的基础仿真。
3. 读懂 UPS 控制固件中的 ISR、任务、状态机、保护和通信模块分工。
4. 面对一个控制或保护问题，能提出“先看什么波形、改哪个参数、如何复现实验”的调试思路。
5. 对每个学过的核心概念，都能从物理直觉、数学形式、仿真现象和工程落地四个角度说明。
